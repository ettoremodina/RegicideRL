ISMCTS Explained

Information Set Monte Carlo Tree Search for Imperfect Information Games
Reference: "Information Set Monte Carlo Tree Search" by Peter I. Cowling, Edward J. Powley, and Daniel Whitehouse.
IEEE Transactions on Computational Intelligence and AI in Games (Volume: 4, Issue: 2, June 2012)

The Problem with Perfect Information

Standard Monte Carlo Tree Search (MCTS) assumes perfect information—meaning the entire state of the game (like a chessboard) is known to all players. In games like Regicide or Poker, the game state contains hidden information (e.g., the deck order).

A naive approach is Perfect Information Monte Carlo (PIMC): randomly guess the hidden cards, run standard MCTS, and average the results. However, this fails due to Strategy Fusion. PIMC implicitly assumes that the agent will know the randomly guessed cards in the future, allowing it to "fuse" separate strategies that would be impossible to execute in reality (because you don't actually know those cards).

The Information Set

Instead of building a tree of absolute game states, ISMCTS builds a tree of Information Sets. An Information Set is simply the perspective of the player: it groups together all possible actual game states that are consistent with what the player has observed so far.

In Regicide, your Information Set consists of: your hand, the discard pile, the current enemy, and the knowledge of which enemies have been defeated. The unknown variables (Tavern Deck order, Castle Deck tier orders) are randomly determinized for each iteration, but the search tree itself only branches on actions taken from the Information Set.

At the start of every iteration, the algorithm samples a random Determinization (it shuffles the hidden decks). It then proceeds through the 4 phases using that specific, fixed determinization.

The Mathematics: The Subset-Armed Bandit

During the Selection phase, the algorithm must choose which child node to explore. Standard MCTS uses the UCB1 formula.

$$ \text{Standard UCB1} = \frac{v_i}{n_i} + C \sqrt{\frac{\ln N}{n_i}} $$

Where \( N \) is the parent node's visit count, and \( n_i \) is the child node's visit count. However, in ISMCTS, we encounter the Subset-Armed Bandit Problem. Because the hidden state is randomized every iteration, an action \( i \) might be legal in one determinization but illegal in another.

If we use standard UCB1, actions that are rarely legal will have a low \( n_i \). As the parent visit count \( N \) grows, the exploration term \( \sqrt{\ln N / n_i} \) will explode to infinity, forcing the algorithm to aggressively try to select an illegal action.

The ISMCTS UCB1 Adaptation

Cowling et al. solved this by replacing the parent's total visit count \( N \) with the action's Availability Count \( A_i \). This tracks how many times action \( i \) was actually legal when the parent node was visited.

$$ \text{ISMCTS UCB1} = \frac{v_i}{n_i} + C \sqrt{\frac{\ln A_i}{n_i}} $$
  • \( v_i \): The total reward accumulated by action \( i \).
  • \( n_i \): The number of times action \( i \) was selected.
  • \( A_i \): The number of times action \( i \) was available (legal) in the sampled determinization.
  • \( C \): The exploration constant (tuning parameter).

By using \( A_i \), the exploration term only grows when the action is actually legal but ignored, correctly balancing exploration and exploitation without breaking the math on illegal moves.

Why This Ensures Your Agent Isn't Cheating

Your agent uses a Single-Observer ISMCTS (SO-ISMCTS). Because it completely shuffles the Tavern Deck and Castle Deck tiers before running the 4 phases on each iteration, the MCTS tree is being built over an aggregate of hundreds of randomly shuffled universes. The algorithm mathematically converges on the play that has the highest probability of success across all possible deck configurations, completely blind to the actual true order of the cards.
