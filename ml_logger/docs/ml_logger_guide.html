ml_logger / guida visuale

Una run.
Una timeline.
Più viste.

Il logger raccoglie log, metriche, parametri, artifact e telemetria attraverso una piccola API stabile. Il progetto cambia; il core rimane.

REGOLA DI CONFINE

Tutto ciò che è generico vive in ml_logger. Tutto ciò che appartiene al dominio vive in un adapter del progetto.

01 / come funziona

Un ingresso comune, destinazioni indipendenti

Il codice applicativo pubblica eventi. Storage, dashboard e report li consumano senza conoscere il training loop.

Codice ML
  • logger.info()
  • log_metrics()
  • log_artifact()
→
RunContext
  • lifecycle della run
  • filtri e configurazione
  • EventBus interno
→
Output
  • SQLite e JSONL
  • dashboard terminale
  • report HTML
  • artifact e telemetria
START

1. Apri una run

Nome, tipo, configurazione e metadati definiscono il contesto dell’esperimento.

STREAM

2. Pubblica eventi

Ogni evento viene prima salvato e poi inviato alle viste attive.

END

3. Finalizza

Il context manager chiude servizi, export e stato della run, anche in caso di errore.

Perché è riutilizzabile: una nuova visualizzazione ascolta gli stessi eventi. Non richiede modifiche al training né allo storage.

02 / come usarlo

Due import e un context manager

run gestisce il lifecycle; get_logger restituisce un normale logger Python.

installazione terminale 
python -m pip install -e .
training.py API consigliata 
from ml_logger import get_logger, run

logger = get_logger(__name__)

with run(
    "training",
    name="baseline",
    config={"learning_rate": 1e-3},
) as experiment:
    experiment.log_params({
        "optimizer": "adam",
        "seed": 42,
    })

    for epoch in range(10):
        loss = train_one_epoch()
        accuracy = evaluate()

        experiment.log_metrics(epoch, {
            "train/loss": loss,
            "eval/accuracy": accuracy,
        })
        experiment.log_progress(
            epoch + 1, 10, "Training"
        )
        logger.info(
            "Epoch %d completata", epoch
        )

    experiment.log_artifact(
        "model.pt", kind="model"
    )

Cosa avviene automaticamente

  1. Inizializzazione Cartelle, manifest, catalogo, handler e viste.
  2. Esecuzione Log, metriche e telemetria condividono la stessa run.
  3. Successo Stato completed, export JSONL e report.
  4. Eccezione Stato failed, errore registrato e risorse chiuse.
Firma corretta: log_metrics(step, metrics_dict). Registra insieme le metriche relative allo stesso step.

04 / configurazione

Scegli cosa salvare e cosa mostrare

I filtri di persistenza e quelli della dashboard sono separati: nascondere una metrica non significa necessariamente perderla.

logger_config.yaml esempio completo ridotto 
logging:
  enabled: true
  level: INFO
  console: true
  file: true

saving:
  enabled: true
  params: true
  metrics: true
  results: true
  artifacts: true
  telemetry: true

metrics:
  include: ["*"]
  exclude: ["debug/*"]

dashboard:
  mode: auto
  refresh_rate: 4
  max_log_lines: 50
  max_metrics: 16
  screen: true
  metrics:
    include:
      - "self_play/*"
      - "train/*"
      - "eval/*"
    exclude: ["debug/*"]

telemetry:
  enabled: true
  sample_interval_sec: 5

report:
  enabled: true
  visualization: auto

Ordine di risoluzione

  1. Parametro esplicito logger_config_path passato a run.
  2. Variabile d’ambiente ML_LOGGER_CONFIG.
  3. File del progetto logger_config.yaml nella directory corrente.
  4. Default del package Configurazione distribuita con ml_logger.
  5. Override del workflow run_type_overrides.<run_type> viene applicato per ultimo.
Impostazione Controlla Valori tipici
dashboard.mode Tipo di visualizzazione terminale auto, live, compact, off
saving.* Categorie persistite boolean per metriche, risultati, artifact e telemetria
metrics.include/exclude Metriche salvate pattern fnmatch, per esempio train/*
dashboard.metrics Metriche visibili nella box live filtri indipendenti dalla visualizzazione del report
report.visualization Grafici nel report finale auto, line, table

05 / funzionalità

API piccola, output prevedibili

SQLite è la fonte canonica. JSONL e report sono output portabili costruiti dalla stessa timeline di eventi.

API A cosa serve Dove arriva
get_logger(name) Log applicativi standard Console, Log Stream, logs/run.log
log_params(dict) Iperparametri e setup Manifest ed eventi SQLite
log_metrics(step, dict) Misure nel tempo Dashboard, SQLite, JSONL e report
log_progress(done, total, label) Avanzamento frequente Footer live, non persistito
log_artifact(path, kind) Modelli, immagini e dataset Copia nella run e catalogo
save_result(name, data) Risultati JSON strutturati File nella run ed evento
subscribe(view) Vista o integrazione personalizzata EventBus in-process

Layout della run

artifacts/
├── catalog.sqlite
└── runs/YYYY-MM-DD/<run-id>/
    ├── manifest.json
    ├── config.json
    ├── logs/run.log
    ├── metrics/metrics.jsonl
    ├── metrics/telemetry.jsonl
    ├── artifacts/
    └── reports/run_report.html

Multiprocessing

# Processo padre
descriptor = experiment.descriptor()

# Worker
worker = RunContext.attach(**descriptor)
worker.log_metrics(step, {
    "worker/throughput": value,
})

I worker scrivono su SQLite WAL. Il padre materializza i JSONL alla chiusura, evitando append concorrenti sullo stesso file.

06 / adattamento

Il progetto traduce; il logger registra

L’adapter converte concetti specifici del dominio in metriche, artifact o eventi senza contaminare il package generico.

PROGETTO

Concetti del dominio

  • partita o episodio
  • dataset split
  • checkpoint promosso
  • confusion matrix
  • replay specifico
→
ML_LOGGER

Primitivi generici

  • metriche e parametri
  • artifact e risultati
  • log applicativi
  • eventi e viste
  • lifecycle della run

Struttura consigliata

project/
├── integrations/
│   └── project_logging.py
├── logger_config.yaml
└── training.py

In Regicide questo ruolo è svolto da integrations/regicide_logging.py.

Adapter minimo

class DatasetRecorder:
    def __init__(self, context):
        self.context = context

    def save_split(self, path, split, rows):
        self.context.log_metrics(rows, {
            f"dataset/{split}/rows": rows,
        })
        return self.context.log_artifact(
            path,
            kind=f"dataset-{split}",
        )

Checklist per un nuovo progetto

  1. Installa il package senza aggiungere classi del dominio.
  2. Crea logger_config.yaml nella nuova repository.
  3. Apri una sola run nel comando principale.
  4. Definisci nomi metrici gerarchici prima del training.
  5. Pubblica valori intermedi durante le fasi lunghe.
  6. Crea un adapter solo per ciò che non è già una metrica o un artifact.
